# 研究方法论指南

本文件规范 `research` 命令的信息收集流程。核心原则：**反复搜索、多方验证、信息溯源**。

---

## Phase 1：官网爬取

### 必须访问的页面

按优先级依次尝试：

| 优先级 | 页面类型 | 常见路径 | 提取目标 |
|--------|----------|----------|----------|
| P0 | 首页 | `/` | 产品名称、Slogan、核心价值主张 |
| P0 | 功能/产品页 | `/features`, `/product`, `/solutions`, `/platform` | 功能列表、差异化卖点 |
| P1 | 定价页 | `/pricing`, `/plans` | 定价模型、套餐对比 |
| P1 | 关于页 | `/about`, `/about-us`, `/company` | 公司背景、团队、使命 |
| P2 | 博客/新闻 | `/blog`, `/news`, `/press` | 最新动态、产品更新 |
| P2 | 客户案例 | `/customers`, `/case-studies` | 目标客户画像、使用场景 |

### 爬取规范

1. **先用 `read_url_content`**，速度快，适合大多数页面
2. **如果返回内容明显不完整**（少于 200 字或明显缺少关键内容），切换到 `browser_subagent` 重新访问
3. **记录每个页面的 URL 和提取到的内容**，存入 `research_state.md`
4. **不要猜测页面路径**，如果 `/features` 不存在，不要假设功能列表内容

### 信息提取清单

从官网页面中提取以下信息（能找到多少算多少）：

- [ ] 产品正式名称
- [ ] 一句话描述 / Slogan
- [ ] 核心功能列表（至少 3 个）
- [ ] 目标用户描述
- [ ] 定价模型和价格
- [ ] 公司名称、创立时间、总部位置
- [ ] 团队规模（如果公开）
- [ ] 技术栈线索（footer、job listings 中可能有）

## Phase 2：联网搜索扩展

### 搜索策略：至少 3 轮

**关键原则**：不要只搜一次就停。每轮搜索的结果会启发下一轮的关键词。

#### 第 1 轮：基础认知（必须）

用英文搜索获取更多信息量，中文搜索作为补充。

```
search_web("{产品名} what is product overview")
search_web("{产品名} review 2025 2026")
search_web("{产品名} pricing plans")
```

**从第 1 轮提取**：
- 产品定位的第三方描述
- 竞品名称（评测文章中通常会提到同类产品）
- 用户评价趋势

#### 第 2 轮：深度扩展（必须）

基于第 1 轮发现的竞品和关键词，进行针对性搜索。

```
search_web("{产品名} vs {竞品A} vs {竞品B}")
search_web("{产品名} funding valuation investors")
search_web("{产品名} target audience use cases")
search_web("{产品名} 用户评价 怎么样")
```

**从第 2 轮提取**：
- 竞品对比信息
- 融资和估值信息
- 目标用户的具体画像
- 中文生态中的信息

#### 第 3 轮：查漏补缺（必须）

检查 `research_state.md` 的研究完成度表格，对 ❌ 和 ⚠️ 的维度进行针对性搜索。

```
# 如果商业模式不清晰
search_web("{产品名} business model how does it make money revenue")

# 如果技术栈不明
search_web("{产品名} tech stack built with technology")

# 如果用户评价不足
search_web("{产品名} G2 reviews Capterra")
search_web("{产品名} 知乎 评价")
```

#### 可选第 4 轮：深度补充

如果前 3 轮后仍有关键维度缺失，可以尝试：

```
search_web("{产品名} CEO interview founder story")
search_web("{产品名} product hunt launch")
search_web("{产品名} crunchbase")
```

### 搜索结果处理规范

1. **记录来源**：每条搜索结果记录 URL 和摘要
2. **交叉验证**：同一信息如果只有一个来源，标记 ⚠️；2+ 来源标记 ✅
3. **时效性**：优先采信 12 个月内的信息；超过 24 个月的信息标注日期
4. **语言**：英文搜索为主（信息量大），中文搜索为辅（本地化视角）

## Phase 3：信息整合与保存

### 写入 research_state.md

将所有收集到的信息按照 `SKILL.md` 中定义的 `research_state.md` 格式写入。

### 完成度自检

填写研究完成度表格，确认 8 个维度的状态：

| 维度 | 判断标准 |
|------|----------|
| 产品名称与描述 | ✅ = 有官网来源；⚠️ = 仅第三方描述；❌ = 未找到 |
| 核心功能列表 | ✅ = 官网有功能页；⚠️ = 从描述推断；❌ = 无信息 |
| 目标用户画像 | ✅ = 官网/案例中明确；⚠️ = 从评测推断；❌ = 无信息 |
| 定价模型 | ✅ = 定价页有价格；⚠️ = 仅知免费/付费；❌ = 无信息 |
| 竞品信息 | ✅ = 2+ 竞品已识别；⚠️ = 仅 1 个；❌ = 无信息 |
| 商业模式 | ✅ = 明确收费方式；⚠️ = 推测；❌ = 无信息 |
| 用户评价 | ✅ = 有评测/评论；⚠️ = 仅少量；❌ = 无信息 |
| 团队/公司 | ✅ = 有公司信息；⚠️ = 仅名称；❌ = 无信息 |

### 信息不足时的处理

如果 ✅ 标记少于 6 个：
1. 告知用户："当前研究信息覆盖 X/8 维度，以下维度信息不足：[列表]"
2. 建议："你可以 (1) 手动提供信息 (2) 尝试补充搜索 (3) 接受现有信息继续分析"
3. 不阻塞流程，但在后续 `analyze` 和 `generate` 中明确标注缺失维度

## 反模式清单

以下行为**严格禁止**：

| ❌ 禁止 | ✅ 正确做法 |
|---------|------------|
| 编造产品不存在的功能 | 只列出有来源的功能，找不到就写"未找到" |
| 猜测定价数字 | 只写官网公开的价格，没有就标"未公开" |
| 编造用户评价 | 只引用搜索到的真实评价 |
| 假装知道竞品 | 只列出搜索结果中出现的同类产品 |
| 编造融资信息 | 仅引用 Crunchbase 等可靠来源 |
| 一次搜索就下结论 | 至少 3 轮搜索，逐步深入 |
| 忽略搜索无果的维度 | 明确标注 ❌ 并在 PRD 中留白 |
